# 비전 — 조직 학습형 디지털 운영자

---

## 이 시스템을 왜 만들었는가

### 풀고자 하는 문제

회사는 시간이 지나도 같은 실수를 반복한다.

- 사람이 떠나면 그 사람이 얻은 교훈도 함께 떠난다
- 같은 종류의 프로젝트를 할 때마다 처음부터 다시 시작한다
- "우리가 왜 그때 그 결정을 했지?"를 아무도 모른다
- 좋은 방법을 발견해도 다음 프로젝트에서 안 쓴다

단순히 "질문에 답하는 AI"나 "문서를 유리하게 업로드하는 도구"로는 해결되지 않는다.
Know-how는 문서에 있지 않다. 사람의 머릿속에 있다.

### 핵심 아이디어

> **실행할수록 학습하는 디지털 직원을 만든다.**

월보관이 쉬워지는 것이 목표가 아니다.
프로젝트를 수행할수록 **조직의 기억과 능력이 컵이는 것**이 목표다.

---

## 이 시스템이 하는 일

### 실행할 때
1. 과거 유사 프로젝트의 Episode를 찾는다
2. Drive에서 관련 원본 문서를 확인한다
3. 실제 결과물을 만든다 (보고서, 분석, 설계 등)
4. 출처를 모두 기록한다

### 평가할 때
1. Executor와 **독립된 관점**으로 결과를 살편다
2. 문제를 합리화하지 않고 명확히 지적한다
3. Pass / Revise / Fail로 판정한다

### 학습을 추출할 때
1. 뭐가 잘 됐고, 왜 잘 됐는지
2. 뭐가 실패했고, 월인이 뭐였는지
3. 다음에 쓸 수 있는 지식을 Episode로 정제한다

### Skill로 진화할 때
1. 반복된 패턴이 있는지 Episode에서 확인한다
2. Skill 개선 **제안**을 생성한다 (Skill을 직접 수정하지 않는다)
3. 사람이 검토하고 승인하면 Skill에 반영된다

## 설계에서 내린 교소

### 왜 Execute와 Evaluate가 분리되어 있는가

실행자가 스스로 평가하면 문제를 합리화한다.
"조금 부족하지만 주어진 시간 내에는 최선이었다"는 판단이 다음 프로젝트로 이어지지 않는다.
독립적인 Evaluator만이 "이 수준으론 목표가 달성되지 않았다"를 명확히 지적할 수 있다.

### 왜 Refine에서 AI가 Skill을 직접 수정하지 않는가

Skill은 이 시스템의 **실제 동작을 바꾸는** 코드다.
검증 안 된 패턴이 자동으로 Skill에 반영되면, 모든 다음 프로젝트에 영향을 준다.
Episode 두 개 이상의 작동 근거, 회귀평가 계획이 있어야 승인한다.
제안과 승인을 분리해야 안전하게 진화할 수 있다.

### 왜 Source of Truth를 4개로 나누었는가

| 시스템 | 역할 | 왜 따로 |
|---|---|---|
| Google Drive | 사람이 만든 원본 | AI가 덮어쓸 수 없는 진짜 원본 |
| GBrain + Supabase | AI가 학습한 조직 기억 | Drive를 대체하지 않음 — 소화된 지식만 |
| GitHub | AI의 업무 능력과 정책 | 회사 기록이 들어오면 안 됨 |
| Claude Code | 사용자 단일 인터페이스 | 유일한 창구 — 내부 연결은 시스템이 알아서 |

Drive에 있는 파일을 GBrain에 통째로 복사하지 않는다.
GBrain에는 Drive를 읽고 얻은 **결론, 판단, 패턴**만 남긴다.

### 왜 Core와 Company Instance를 분리하는가

나중에 회사 B, C에돆돆 배포할 계획이다. 각 회사의 기밀이 섭이면 안 된다.
Core는 **능력(학습 루프, Skill 구조, 평가 기준)**를 공유하고,
Company Instance는 **기억(경험, 의사결정, 노하우)**을 가직 각자 구체화한다.

## 성공의 정의

문서 검색 횟수나 AI 답변 속도가 아니다. 다음으로 판단한다:

| 지표 | 의미 |
|---|---|
| **과거 Episode 재사용률 ≥50%** | AI가 배운 것을 실제로 다음에 활용하는가 |
| **동일 실패 반복 <10%** | 조직이 진짜로 더 똑똑해지고 있는가 |
| **평가 Pass률 ≥70%** | 결과물 품질이 유지되는가 |
| **출처 추적 100%** | 모든 주장이 원본으로 연결되는가 |
| **Skill 제안 품질 ≥30%** | 제안한 Skill이 승인되어 실제 성과를 내는가 |

## 현실적 역할 (오늘의 나)

완전자율 CEO가 아니다. 지금은 다음에 가깝다:

- **Chief of Staff**: 업무 흐름을 접수하고 조율
- **프로젝트 리더**: 결과물을 실제로 생산
- **리서치/의사결정 지원**: 판단의 근거 제공
- **학습 담당자**: 경험을 컵고 능력으로 진화

다음은 반드시 사람이 한다:
확정 계약, 비용 집행, 인사 결정, 외부 이메일 발송, 데이터 삭제, 운영 Skill 변경, Drive 원본 수정, 회사 기밀의 외부 API 전송.

## 다회사 확장 비전

```
digital-operator-core     ← 공통 업무 능력 (Stateless)
company-a-operator        ← 회사 A 전용 기억과 정책
company-b-operator        ← 회사 B 전용 (미래)
company-c-operator        ← 회사 C 전용 (미래)
```

각 회사의 기밀은 절대 섭이지 않는다.
하지만 "어떻게 시장을 분석하는가", "어떻게 의사결정을 구조화하는가"는 Core를 통해 모든 회사가 혜택을 받는다.
